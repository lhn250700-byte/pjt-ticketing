# 대규모 트래픽 대응 선착순 티켓 예매 시스템

대규모 트래픽(Peak Time) 상황에서 서버 자원 고립을 방지하고 데이터 무결성을 보장하기 위해, Redis 대기열 아키텍처와 Kafka 분산 이벤트 체이닝(Event Chaining)을 결합하여 구현한 고성능 비동기 티켓팅 시스템입니다.

<br>

## Tech Stack
- **Backend**: Java 17, Spring Boot, Spring Data JPA
- **Database & Caching**: MySQL, Redis (Lua Script)
- **Message Queue**: Apache Kafka
- **Performance Testing**: k6

<br>

## Architecture Evolution

### [As-Is] 단일 이벤트 비동기 처리 구조
- **흐름**: API 서버가 Redis 가선점 및 RDB 내 예약 데이터를 PENDING 상태로 적재 후 202 Accepted를 반환하며, 단일 Kafka 토픽으로 통합 이벤트를 발행하는 방식.
- **한계**: 단일 Consumer가 결제 검증, 예약 확정, 좌석 점유 등 무거운 비즈니스 로직을 모두 처리함. 외부 PG사 연동 등 특정 구간에서 병목이 발생할 때 해당 세그먼트만 독립적으로 수평 확장(Scale-out)할 수 없는 아키텍처적 한계 존재.

### [To-Be] 3단계 분산 이벤트 체이닝 아키텍처 (현재)
- **개선**: 기존 단일 토픽 구조를 독립된 3개의 토픽 및 Consumer 구조로 분리하여 릴레이 형태의 비동기 파이프라인 구축.
- **이점**: 컴포넌트 간 결합도를 낮추고, 단계별 예외 처리 기반의 보상 트랜잭션을 설계하여 분산 환경에서의 실시간 데이터 정합성과 개별 컴포넌트의 수평 확장성 확보.

<br>

## 데이터 흐름 (Data Flow)

1. **대기열 진입 및 순번 부여**
   - 유저는 Redis Sorted Set 기반의 Waiting Queue에 진입하여 순번을 부여받고 대기 상태 유지.
2. **진입 허가 및 토큰 전환**
   - 차례가 도래한 유저의 토큰은 Redis String 기반의 Active Key로 전환되어 예약 권한 획득.
3. **비동기 접수 및 파이프라인 트리거 (202 Accepted)**
   - 예약 요청 시 API 서버는 Redis Lua Script를 통해 좌석 가선점의 원자성(Atomicity)을 검증.
   - 검증 성공 시 DB에 데이터를 PENDING 상태로 경량 적재 후 202 Accepted 응답과 함께 reservationId를 반환하여 커넥션을 빠르게 풀(Pool)에 반환.
4. **백그라운드 이벤트 체이닝 (Kafka 파이프라인)**
   - **Consumer 1 (Payment)**: 이벤트 체이닝 (Kafka 파이프라인)**
   - **Consumer 1 (Payment)**: 이벤트를 수신하여 금액 검증 후 RDB에 결제 데이터 기입. 성공 시 Active 대기열 토큰을 파기하여 대기열 순환 속도 향상.
   - **Consumer 2 (Reservation)**: 결제 완료 이벤트를 받아 예약 상태를 PENDING에서 CONFIRMED로 변경.
   - **Consumer 3 (Seat)**: 최종 RDB 좌석 점유 상태를 반영하고, finally 블록을 통해 Redis에 설정되었던 5분 가선점 자원(Hold Key)을 제거하여 자원 정리.

<br>

## 기술적 의사결정

### 1. 비동기 예약 처리 파이프라인 및 이벤트 체이닝 구축
- **도입 배경**: 외부 PG 연동 등 무거운 I/O 바운드 작업으로 인해 동기식 커넥션 풀 고갈(Starvation)이 발생할 수 있는 문제를 근본적으로 해결하고자 도입.
- **기술 비교**: 대규모 피크 타임 트래픽을 메모리 유실 없이 안정적으로 수용하고 영속성을 보장하기 위해, RabbitMQ 대신 디스크 기반 큐잉인 Kafka를 선택. 프로세스를 [결제 장부 생성 -> 예약 상태 변경 -> 좌석 상태 반영 및 자원 청소]의 3단계 독립 이벤트 구조로 파이프라인화하여 아키텍처적 결합도를 낮추었습니다.
- **장애 격리**: 특정 비즈니스 단계에서 예외 발생 시 다음 단계로의 이벤트 전파를 차단하고, catch 블록 내에서 예약을 FAILED 처리한 뒤 Redis 가선점 자원을 원복하는 보상 트랜잭션 메커니즘을 적용하여 분산 환경에서의 정합성을 보장합니다.

### 2. 분산 파티셔닝을 통한 Consumer 수평 확장성 확보
- **도입 배경**: 대규모 분산 환경에서 메시지의 순서를 보장하면서도 파이프라인의 병렬 처리 능력을 극대화하기 위함입니다.
- **구현 성과**: Kafka의 파티션 분산 메간즘과 Consumer Group을 활용하여 대량의 예약 요청을 순서 오류 없이 처리하도록 구성했습니다. 이벤트 체이닝으로 격리된 3개의 토픽마다 독립적인 Consumer Group을 매핑함으로써, 병목이 발생하는 특정 워커 세그먼트(예: 외부 I/O 병목 구간)의 Consumer 스레드만 독립적으로 Scale-out할 수 있는 유연성을 확보했습니다.

## 부하 테스트 결과 (k6 Performance Test)

동일한 인기 좌석을 대상으로 1,000명의 동시 사용자가 경합하는 상황을 모사하여 아키텍처 검증을 진행했습니다.

### 1. 규모별 성능 지표 변화

> **Scenario**: 1,000명의 VU가 대기열 진입 후 동일한 좌석(ID: 10)에 대해 동시에 예매 요청 송신 (shared-iterations)

| 지표 / 부하 규모 | VU 100 | VU 500 | VU 1000 (Peak) |
| :--- | :--- | :--- | :--- |
| **성공 (status_202)** | 정확히 1건 | 정확히 1건 | 정확히 1건 |
| **경합 실패 (status_409)** | 99건 | 499건 | 999건 |
| **평균 응답 속도 (avg)** | 23.68ms | 51.08ms | 238.59ms |
| **95% 응답 속도 (p95)** | 58.80ms | 139.95ms | 823.09ms |
| **통합 시스템 TPS** | 24.65 req/s | 117.45 req/s | 216.96 req/s |

### 2. 성능 결과 분석
- **동시성 제어 및 데이터 정합성 보장**: 부하 규모에 관계없이 예매 성공(202) 지표는 정확히 1건만 발생했습니다. Redis Lua Script 기반의 선점 원자성 로직이 레이스 컨디션을 제어하여, 최초 선점자 외의 경쟁 트래픽은 RDB에 도달하기 전 409 Conflict로 즉시 Fast-Fail 처리되었음을 확인했습니다.
- **Fast-Fail 구조를 통한 지연 시간 최소화**: 무거운 데이터 영속화 및 비즈니스 로직을 Kafka 비동기 파이프라인으로 이관함으로써, 최고 피크 타임(1,000 VU) 상황에서도 API 서버의 평균 응답 시간을 238.59ms 수준으로 방어했습니다.
- **선형적 확장성(Linear Scalability) 확인**: 트래픽 규모(VU)가 증가함에 따라 시스템 처리 성능(TPS)이 24.65 -> 117.45 -> 216.96으로 정비례하며 우상향 구조를 유지, 대규모 트래픽 환경에서도 시스템 마비 없이 선형적으로 확장 가능한 구조임을 검증했습니다.
